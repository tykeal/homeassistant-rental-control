# Implementation Plan: Fix Duplicate Keymaster Code Slot Assignment

**Branch**: `005-fix-duplicate-slot` | **Date**: 2025-07-18 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/005-fix-duplicate-slot/spec.md`

## Summary

Fix the race condition in rental-control's slot assignment where multiple
calendar sensors reading `EventOverrides.next_slot` during the same coordinator
update cycle can assign the same physical Keymaster code slot to different
guests. The fix introduces an `asyncio.Lock` on `EventOverrides` with an atomic
"reserve-or-get" async method, deduplication enforcement at the storage layer
(name + time-range overlap identity), pre-execution slot verification in utility
functions, and retry/escalation for failed lock commands (3 cycles then
`persistent_notification`).

## Technical Context

**Language/Version**: Python ≥3.13.2 (per `pyproject.toml` `requires-python`)
**Primary Dependencies**: Home Assistant Core (≥2025.x), `pytest-homeassistant-custom-component`, Keymaster integration (lock code management)
**Storage**: In-memory `dict[int, EventOverride | None]` in `EventOverrides._overrides`; Keymaster entity state in HA state machine (survives restarts)
**Testing**: pytest + pytest-asyncio + pytest-homeassistant-custom-component; mypy; ruff; interrogate (100% docstring coverage)
**Target Platform**: Home Assistant (asyncio event loop, Raspberry Pi to x86 server)
**Project Type**: Single HA custom integration
**Performance Goals**: All slot mutations must complete within the coordinator refresh cycle (default 2 min, minimum 30s); no event-loop blocking
**Constraints**: Single-threaded asyncio — concurrency from interleaved coroutines at `await` points only; `@callback`-decorated methods are synchronous (no `await`); Keymaster does not persist custom metadata (UIDs lost on restart)
**Scale/Scope**: 5 managed slots (default), up to ~20 calendar events per refresh; 5 source files modified, ~300 net new LOC estimated
